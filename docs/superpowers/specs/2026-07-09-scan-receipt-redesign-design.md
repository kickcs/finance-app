# Редизайн сканирования чеков (scan-receipt) — дизайн-спека

Дата: 2026-07-09
Статус: реализовано (план: docs/superpowers/plans/2026-07-09-scan-receipt-redesign.md)
Ветка: feat/recipient-scan

## 1. Цель и рамки

Переработать 4-шаговый визард сканирования чека (`frontend/src/features/scan-receipt`,
`frontend/src/pages/scan-receipt/ScanReceiptPage.vue`, `backend/src/modules/receipt`):

- починить приём HEIC с айфона;
- сделать понятными сборы/комиссии и деление позиций (шаг 2);
- упростить управление участниками (шаг 3);
- добавить «Платил не я», мульти-фото, ручной режим, сверку суммы, черновик,
  «прошлую компанию», undo удаления, «остальное — на меня»;
- добавить шаринг чека публичной ссылкой (получателю не нужно приложение) с
  реквизитами «куда переводить» и OG-превью в Telegram;
- полный визуальный проход по всем шагам **строго в существующей дизайн-системе**
  (семантические токены, `shared/ui`, хаптика, reduced-motion).

Не в рамках: QR-код за столом, память места, share-target PWA, Telegram-бот с фото,
история чеков, фото у транзакции, веса позиций, деление без позиций, скидки
(отрицательные сборы), авто-категория, пересчёт валюты, детект дубликатов,
отметка «я оплатил» на публичной странице. Шаринг ссылкой — бесплатная фича
(без PremiumGuard).

## 2. Шаг 1 «Фото» — мульти-фото, ручной режим, HEIC

### 2.1 Композиция

Видоискатель со сканлайном остаётся визуальной подписью. Под рядом
галерея/затвор добавляется текстовая кнопка **«Ввести вручную»** — пропускает OCR
и открывает шаг 2 с одной пустой позицией (валюта — из профиля пользователя,
`useUserCurrency`).

### 2.2 Мульти-фото (до 3 кадров)

Автостарт OCR при выборе фото убирается. Новый флоу:

- выбор фото (камера или галерея; на gallery-input добавляется `multiple`) →
  превью с лентой миниатюр (кадры + тайл «+ Добавить кадр», максимум 3);
- primary-кнопка **«Распознать»** запускает OCR по всем кадрам одним запросом;
- миниатюру можно удалить (крестик) до распознавания;
- дальше как сейчас: оверлей сканирования, успех, автопереход на шаг 2.

`usePhotoStep` переходит с `selectedFile: File | null` на `selectedFiles: File[]`
(и массив preview-URL). Состояния OCR (loading/success/error) не меняются.

Бэкенд: `FileInterceptor('image')` → `FilesInterceptor('image', 3)` — имя поля
прежнее, 1–3 файла, старые клиенты (mobile) продолжают работать.
`ReceiptOcrService.scanReceipt(files: {buffer, mimetype}[])` собирает несколько
`image_url`-частей в одно сообщение; в системный промпт добавляется абзац:
несколько изображений = перекрывающиеся сегменты ОДНОГО чека сверху вниз,
объединять в один список без дублей на стыках. Контракт ответа не меняется.

### 2.3 HEIC

В `processFile` (Step1PhotoCapture):

1. детект HEIC: `file.type` ∈ {`image/heic`, `image/heif`} ИЛИ `type` пустой и
   расширение `.heic/.heif`;
2. валидация «только изображения» пропускает HEIC-по-расширению (сейчас режет);
3. конвертация: сначала существующий нативный путь `<img>`+canvas (Safari
   декодирует HEIC сам); при `img.onerror` на HEIC-файле — ленивый
   `import('heic-to')` (wasm-чанк грузится только при встрече HEIC) → JPEG →
   существующий resize-пайплайн `toJpeg`.

Новая зависимость frontend: `heic-to`. Детект-логика выносится в
`model/imageFile.ts` (чистые функции — покрываются юнитами).

### 2.4 Ошибки OCR

Оверлей ошибки: человеческий заголовок и подсказка («Не получилось прочитать чек —
попробуйте ярче свет или переснимите ближе»), техдетали (`message`, тип, размер) —
свёрнутая мелкая строка `<details>`-паттерном, а не сырой текст поверх фото.

## 3. Шаг 2 «Позиции» — карточки, sheet-редактор, сборы, деление, сверка

### 3.1 Карточка позиции + sheet-редактор

Инлайн-инпуты с мелкими зонами тапа заменяются читаемой карточкой:
название, `qty × цена`, итог строки, цветные точки участников (заполняются на
шаге 3). Тап по карточке открывает **bottom-sheet редактор** (`UModal`):
крупное поле названия, степпер количества, цена за единицу, итог строки
(правка итога пересчитывает цену), кнопки «Разделить» и «Удалить» внутри sheet.
Свайпы влево/вправо (удалить/разделить) и десктопные hover-кнопки сохраняются.
Enter-навигация по полям остаётся внутри sheet.

### 3.2 Undo удаления

Удаление позиции (свайп или из sheet) — мягкое: тост
`toast({ title: 'Позиция удалена', action: { label: 'Вернуть', onClick } })`,
позиция физически удаляется из состояния сразу, «Вернуть» вставляет её обратно
на прежний индекс (включая назначения участников).

### 3.3 Сборы и чаевые

Управление сборами переезжает из футера в явный блок **«Сборы и чаевые»** после
списка позиций:

- каждый сбор — строка: название, значение (`10%` или `7 990 UZS`), переключатель
  вкл/выкл (`UToggle`), тап по значению — инлайн-правка;
- **«+ Добавить сбор»** открывает sheet: пресеты (Обслуживание 10 %, Чаевые,
  Доставка) и свой вариант — название + сегмент-контрол **[ % | сумма ]** +
  числовое поле. Оба типа (`percent`/`amount`) создаются и редактируются из UI
  (модель `ReceiptCharge` это уже умеет; сейчас UI умеет добавлять только процент);
- футер (`TotalFooter`) упрощается до итогов «Позиции + Сборы = Итого» и CTA.

### 3.4 Деление позиции

`SplitItemModal` заменяется sheet «Разделить позицию»:

- заголовок-контекст: «Шашлык · 3 шт · 54 000»;
- режим **«По 1 шт на строку»** — qty N → N строк по 1 шт (лекарство для
  «нечётное количество на меньшее число людей»; доступен при целом qty ≥ 2,
  разумный предел — 10 строк);
- **кастомное деление** — степпер первой части с живым превью обеих строк
  («Строка 1: 2 шт · 36 000 / Строка 2: 1 шт · 18 000»);
- подсказка внизу: «Если блюдо ели вместе — не делите, а отметьте нескольких
  участников на следующем шаге: сумма разделится поровну».

### 3.5 Сверка суммы с чеком

`handleOcrResult` начинает сохранять `result.totalAmount` в новый ref
`ocrTotalAmount` (сейчас значение выбрасывается). На шаге 2, если
`|totalAmount − ocrTotalAmount| / ocrTotalAmount > 0.01`, над футером — warning:
«Позиции дают X, а в чеке итог Y (разница Z)» с действиями
**«Добавить разницу строкой»** (позиция «Прочее», qty 1, цена Z; только при Z > 0)
и «Скрыть» (до конца сессии визарда). В ручном режиме и при отсутствии
`totalAmount` в OCR сверка не показывается.

## 4. Шаг 3 «Участники» — кисть отдельно, управление отдельно

### 4.1 Разделение режимов

- **Панель сверху — только «кисть»**: аватар-чипы выбора активного участника.
  Управление (добавить/удалить/кто платит) из чипов убирается; остаётся кнопка
  «Изменить» → sheet.
- **Sheet «Участники»** — единственное место управления: список строк
  (аватар, имя, селектор «Платит: сам / за него платит [X]», удаление), блок
  добавления: чип «Я» (если нет), чипы контактов из `usePeople`
  (`entities/person`), поле нового имени. Удаление участника с назначениями —
  подтверждение.
- Пустое состояние шага сразу открывает этот sheet.

### 4.2 Прошлая компания

После успешного сабмита состав участников (имена + isMe + paidBy-структура)
сохраняется в `localStorage` (`useLocalStorage`, ключ `scan-receipt:last-party`).
На шаге 3 при пустом списке участников показывается чип
**«Как в прошлый раз: Вы, Аня, Тимур»** — тап добавляет всех разом.

### 4.3 Быстрые действия

Ряд над списком позиций:

- **«Поровну на всех»** — назначает все позиции всем участникам;
- **«Остальное — на меня»** — добирает неназначенные позиции участнику isMe
  (виден, когда есть «Я» и есть неназначенные);
- чип-фильтр **«Без участника (N)»** — фильтрует список до неназначенных;
  warning в футере тапабелен и включает тот же фильтр;
- в строке позиции — мини-действие «все» (назначить/снять позицию всем сразу,
  использует существующий `ALL_PARTICIPANTS_ID`).

## 5. Шаг 4 «Итог» — «Кто платил» и полировка

### 5.1 Кто платил

Новый селектор **«Кто платил»** (по умолчанию «Я») над формой транзакции.
Состояние — `payerId: string | null` в визарде (null = «Я» даже без участника
isMe).

Если платил другой участник X:

- расходная транзакция **не создаётся** (деньги ушли не с моего счёта);
- создаётся один долг `debt_type: 'taken'` («я должен») плательщику X на мою
  долю (итог участника isMe, включая тех, за кого плачу я через paidBy);
- форма упрощается: категория и тумблер `createDebts` скрываются, остаётся счёт
  (нужен для `debtsApi.create`), дата, описание; вместо тумблера — карточка-итог
  «Будет создан долг: вы должны X — 85 000 UZS»;
- если доля «Я» равна 0 или участника isMe нет — сабмит блокируется с подсказкой;
- долги других участников перед X не создаются (это отношения X, не мои);
- в Telegram-import-флоу (`importedId`) селектор фиксируется на «Я» — импорт
  подтверждается только транзакцией.

Ретрай-логика `useSubmitStep` (идемпотентные `createdTransactionId` /
`createdDebtParticipantIds`) распространяется на новый путь.

### 5.2 Полировка

Дерево «кто сколько» остаётся. Форма транзакции и тумблер долгов группируются в
секцию «Параметры записи» с заголовком; ошибки сабмита — в существующем стиле.

## 6. Черновик визарда

Композабл `model/useReceiptDraft.ts`:

- сериализует в `localStorage` (`useLocalStorage`, ключ `scan-receipt:draft`):
  шаг, items, currency, storeName, ocrTotalAmount, charges, participants,
  payerId, formData, savedAt; фото не сохраняется;
- запись — `watchDebounced` (~500 мс), начиная с шага 2 (после OCR/ручного входа);
- при открытии страницы со свежим (< 24 ч) черновиком с позициями — баннер на
  шаге 1 «Продолжить прошлый чек? · N позиций · Итого X» с «Продолжить»
  (восстановление на сохранённый шаг) и «Начать заново» (очистка);
- очистка: успешный сабмит, «Начать заново», протухание.

## 7. Шаринг публичной ссылкой

### 7.1 Бэкенд (модуль receipt)

Новая ORM-сущность `SharedReceiptOrmEntity`, таблица `shared_receipts`
(+ миграция; регистрация в `data-source.ts` И `app.module.ts`):

| колонка      | тип                        |
|--------------|----------------------------|
| id           | uuid PK                    |
| token        | varchar(21) unique (nanoid)|
| user_id      | uuid, индекс               |
| payload      | jsonb                      |
| created_at   | timestamptz                |

`payload` — самодостаточный снапшот: storeName, date, currency, totalAmount,
subtotal, charges (label/value), participants[] (name, color, isMe, total,
paidByName, items[] {name, share, sharedWith, lineTotal}), paymentMethods[]
({label, value}), ownerName. Ссылка бессрочная, «живых» данных не содержит.

Эндпоинты:

- `POST /api/receipts/share` (JWT) — body: payload; ответ `{ token, url }`,
  где `url` строится бэкендом из новой env-переменной `PUBLIC_APP_URL`
  (в dev — `http://localhost:3000`, где `/r/:token` отдаёт сам бэкенд;
  в prod — домен приложения, добавить в `.env.example` и deploy-секреты);
- `GET /api/receipts/shared/:token` (`@Public()`) — JSON payload (404 → NotFound);
- `GET /api/receipts/shared/:token/og.png` (`@Public()`) — OG-картинка:
  SVG-шаблон карточки чека (место, дата, итог, до 4 участников с долями,
  «+N ещё») → PNG через `sharp` (новая зависимость backend; поддержка SVG есть в
  prebuilt-бинарях), `Cache-Control: public, max-age=86400`;
- `GET /r/:token` (`@Public()`, вне префикса `/api` — `@Controller` с абсолютным
  путём + исключение в `setGlobalPrefix`) — мини-HTML: og:title
  («Чек из Кафе — 250 000 UZS»), og:description («4 участника · доли внутри»),
  og:image → `/api/receipts/shared/:token/og.png`, og:url, twitter:card
  `summary_large_image` + мгновенный JS/meta-refresh redirect на SPA
  `/shared/:token`. Бот Telegram читает OG, живой браузер уезжает на SPA.

Инфраструктура: в prod nginx фронтенда должен проксировать `location /r/` на
бэкенд (правка nginx-конфига/`docker-compose.prod.yml` — включить в план;
в dev проксирование не нужно — ссылка ведёт напрямую на бэкенд).

Экранирование: все строки payload экранируются в HTML/SVG-шаблонах
(имена участников и магазина — пользовательский ввод; XSS/SVG-инъекция).

### 7.2 Сохранённые реквизиты

Новая сущность `PaymentMethodOrmEntity`, таблица `payment_methods`
(id, user_id, label, value, created_at) в контексте identity + миграция.
Эндпоинты (JWT): `GET /api/payment-methods`, `POST` (label ≤ 50, value ≤ 100,
лимит 10 шт.), `DELETE /api/payment-methods/:id`.
Сохранение карты — **только по явной галочке** пользователя (по умолчанию
выключена): приватность — его выбор. Значение хранится как введено (это
реквизиты для перевода, не PCI-данные — но в UI подписать «видно всем, у кого
есть ссылка»).

Фронтенд: `entities/payment-method` по стандартному паттерну
(`paymentMethodApi.ts`, `usePaymentMethods.ts`, `queryKeys.ts`, `model/types.ts`).

### 7.3 UX шаринга (экран успеха)

На `SuccessOverlay` primary-действие **«Поделиться ссылкой»** (существующие
«Картинкой/Текстом/Сохранить» остаются). Sheet:

1. секция «Куда переводить»: чипы сохранённых карт (мультивыбор),
   «+ Добавить реквизиты» (название + значение + галочка «Сохранить для будущих
   чеков», по умолчанию выкл.), можно вообще без реквизитов;
2. «Создать ссылку» → `POST /api/receipts/share` → ссылка `https://<origin>/r/<token>`
   с кнопками «Копировать» (clipboard + toast) и системный share
   (`navigator.share`, фолбэк — копирование).

Повторное открытие sheet после создания показывает готовую ссылку (токен
хранится в состоянии визарда до ухода со страницы).

### 7.4 Публичная страница `/shared/:token`

SPA-роут без auth-guard (`meta: { public: true }`), фича
`features/view-shared-receipt` + страница `pages/shared-receipt`:

- карточка-«чек» в дизайн-системе (тот же скеоморфный стиль, что
  `ReceiptTotalCard`): место, дата, итог, сборы;
- разбивка по участникам с раскрытием позиций (по образцу шага 4, read-only);
- блок «Куда переводить»: реквизиты с кнопкой копирования у каждой строки;
- футер «Создано в <приложение>» со ссылкой на лендинг/регистрацию
  (виральная механика);
- состояния: загрузка (skeleton), 404 («Чек не найден или ссылка неверна»),
  ошибка сети (retry). Тёмная/светлая тема работают.

## 8. Данные и потоки (сводка изменений модели)

`useReceiptWizard` дополняется: `ocrTotalAmount`, `payerId`, `manualMode`
(вход без OCR), интеграция `useReceiptDraft`, «прошлая компания», undo-буфер
удалённой позиции. Пропсы/эмиты шагов расширяются соответственно; API-слой
фронта: `receiptApi.scan(files: File[])`, `receiptApi.share(payload)`,
`sharedReceiptApi.get(token)` (публичный fetch без токена).

## 9. Ошибки и краевые случаи

- HEIC не сконвертировался (битый файл) — та же человеческая ошибка формата;
- мульти-фото: часть кадров нечитаема — OCR вернёт что смог, сверка суммы
  подсветит расхождение;
- удаление участника-плательщика (`payerId`) — payerId сбрасывается на «Я»;
- удаление участника, на которого ссылались paidById — уже обнуляется (есть);
- черновик от старой версии структуры — try/parse + отбрасывание при
  несовпадении схемы (версионное поле `v: 1`);
- публичная страница: токен не найден → 404-экран; OG-эндпоинт при отсутствии
  токена → 404 без картинки;
- `sharp` недоступен/упал — `/r/:token` отдаёт OG без og:image (текстовое
  превью), страница работает.

## 10. Тестирование

- `useReceiptWizard.spec.ts` расширяется: сверка суммы (`ocrTotalAmount`,
  добавление разницы), ручной режим, payerId-логика сабмита («платил не я» →
  нет транзакции, один долг taken), undo удаления, «прошлая компания»,
  черновик (сохранение/восстановление/протухание);
- юниты на `model/imageFile.ts` (детект HEIC по type/расширению);
- backend: юнит на multi-image сборку сообщения OCR (мокнутый OpenAI), e2e/юнит
  на share-эндпоинты (создание, публичное чтение, 404, экранирование в OG-HTML);
- ручная проверка: HEIC с реального айфона, Telegram-превью ссылки,
  тёмная тема публичной страницы, reduced-motion.

## 11. Изменения вне фичи

- `frontend/src/app/router/index.ts` — публичный роут `/shared/:token`;
- nginx prod-конфиг — `location /r/` → backend;
- `backend/src/main.ts` — исключение `/r` из глобального префикса;
- `data-source.ts` + `app.module.ts` — регистрация двух новых ORM-сущностей;
- две миграции (`shared_receipts`, `payment_methods`);
- зависимости: frontend `heic-to`, `nanoid` (если ещё нет); backend `sharp`, `nanoid`;
