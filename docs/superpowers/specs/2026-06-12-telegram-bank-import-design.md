# Импорт банковских уведомлений через Telegram-бота

**Дата:** 2026-06-12
**Статус:** утверждён пользователем

## 1. Контекст и цель

Банковские боты в Telegram (HUMO-карты, UZS) присылают пользователям уведомления об операциях в личный чат. Перехватить этот чат автоматически нельзя: userbot отвергнут пользователем, Telegram Business-боты не получают сообщения других ботов (ограничение Bot API), перехват уведомлений на устройстве создаёт зависимость от телефона.

**Решение:** общий Telegram-бот приложения, которому пользователи вручную форвардят уведомления банка (можно пачками). Бот парсит сообщения и складывает их в инбокс «На подтверждение». Транзакция попадает в приложение только после прохождения флоу подтверждения, где пользователь указывает счёт, категорию, при необходимости — долг, деление на людей, скан чека.

**Ключевые решения:**
- Фича мультипользовательская: один бот на всех, привязка аккаунта по deep link
- Фича бесплатная (без PremiumGuard)
- Импорт идёт в инбокс на проверку, не создаёт транзакции автоматически
- Бот живёт внутри существующего NestJS backend (webhook), отдельных процессов нет

## 2. Поддерживаемые форматы сообщений (v1)

Парсер — реестр (`canParse(text)` / `parse(text)`), v1 включает один источник (HUMO-бот) с тремя типами:

### 2.1 Оплата → расход (`expense`)

```
💸 Оплата
➖ 1.700,00 UZS
📍 TRANSPORT TOLOV>TOS
💳 HUMOCARD *1951
🕓 22:11 12.06.2026
💰 12.543.101,08 UZS
```

### 2.2 Пополнение → доход (`income`)

```
🎉 Пополнение
➕ 103.500,00 UZS
📍 HAMKOR HUMO P2P>Andi
💳 HUMOCARD *1951
🕓 23:35 11.06.2026
💰 887.801,08 UZS
```

### 2.3 Смена баланса (зарплата) → доход (`balance_change`)

```
ℹ️ Счет по карте изменен
💸 13.244.800,00 UZS
💳 HUMO-CARD *1951
🕘 15:39 12.06.2026
```

Здесь `💸` — **новый баланс карты**, суммы операции в сообщении нет. По словам пользователя, такие сообщения означают зачисление зарплаты. Сумма дохода вычисляется как дельта: `новый баланс − последний известный balance_after по той же карте этого пользователя`. Если предыдущего баланса нет — сумма остаётся пустой, пользователь вводит её в флоу подтверждения. Если дельта отрицательная — предлагаем тип «расход».

### 2.4 Правила парсинга

- **Числа:** точка — разделитель тысяч, запятая — десятичный (`12.543.101,08` → `12543101.08`)
- **Дата:** `HH:mm DD.MM.YYYY`, таймзона Asia/Tashkent (UTC+5)
- **Карта:** нормализуем `HUMOCARD *1951` / `HUMO-CARD *1951` → `*1951`; вариации написания бренда и эмодзи часов (`🕓`/`🕘`) не должны ломать парсинг
- **Мерчант:** строка `📍`, может отсутствовать (тип 2.3)
- **Тип:** определяется по первой строке (`💸 Оплата` / `🎉 Пополнение` / `ℹ️ Счет по карте изменен`)
- Нераспознанные сообщения сохраняются с raw-текстом и флагом `unparsed` (для добавления будущих форматов), бот отвечает «не смог распознать»

### 2.5 Дедупликация

`dedup_hash = sha256(card_mask | type | occurred_at | amount | balance_after)`, уникальный индекс `(user_id, dedup_hash)`. Для `unparsed`-записей `dedup_hash = sha256(raw_text)`. Повторный форвард молча пропускается — это делает безопасным форвард пачками и заливку истории.

## 3. Backend

Новый bounded context `backend/src/modules/telegram-import/` (структура domain → application → infrastructure → presentation, как у остальных модулей). Библиотека бота — **grammY** в webhook-режиме.

### 3.1 Webhook

- `POST /api/telegram-import/webhook` — `@Public()`, проверка заголовка `X-Telegram-Bot-Api-Secret-Token` (паттерн LemonSqueezy)
- На старте приложения идемпотентный `setWebhook` (URL из env)
- Бот обрабатывает только личные чаты; группы/каналы игнорирует

### 3.2 Привязка аккаунта

1. `POST /api/telegram-import/link-token` (JWT) → генерирует одноразовый токен (TTL 15 минут, таблица `telegram_link_tokens`), возвращает `{ deepLink: "https://t.me/<bot>?start=<token>" }`
2. Бот на `/start <token>`: валидирует токен, создаёт связь `telegram_user_id ↔ user_id`, отвечает подтверждением и инструкцией
3. `GET /api/telegram-import/link` (JWT) → `{ linked, telegramUsername }`
4. `DELETE /api/telegram-import/link` (JWT) → отвязка

Ограничения: один Telegram-аккаунт ↔ один аккаунт приложения (unique на обе колонки). `/start` без токена → инструкция и статус привязки. Форвард от непривязанного пользователя → ответ со ссылкой на привязку в приложении.

### 3.3 Приём сообщений

Текст сообщения (форвард или прямая вставка текста — не различаем) → реестр парсеров → создание `imported_transaction` со статусом `pending`. Ответы бота агрегируются: при пачке форвардов копим результаты ~3 секунды и отвечаем одной сводкой «✅ Импортировано 12 · ⏭ дублей 3 · ⚠️ не распознано 1».

### 3.4 Инбокс и подтверждение

- `GET /api/telegram-import/inbox` (JWT) → pending-элементы **кроме `unparsed`** + `suggestedAccountId` (join с `card_account_mappings`); отдаём также счётчик
- `POST /api/telegram-import/inbox/:id/confirm` (JWT), body `{ transactionId }` — помечает `confirmed`, сохраняет ссылку на транзакцию, upsert'ит маппинг `card_mask → account_id` по счёту созданной транзакции
- `POST /api/telegram-import/inbox/:id/dismiss` (JWT) — помечает `dismissed`

**Оркестрация подтверждения — на фронтенде** (вариант B): страница подтверждения переиспользует существующие механизмы создания транзакции, сплита (транзакция + N долгов через `source_transaction_id`, сначала транзакция — потом долги), долга и чека; после успешного создания вызывает `confirm` с `transactionId`. Это сохраняет консистентность с текущими флоу (split-expense, scan-receipt) и не дублирует их логику на сервере.

### 3.5 Данные (новые таблицы)

```
telegram_links:        id, user_id (unique, FK), telegram_user_id (unique),
                       telegram_username, created_at

telegram_link_tokens:  id, user_id, token, expires_at, used_at, created_at

imported_transactions: id, user_id (FK), raw_text,
                       type ('expense'|'income'|'balance_change'|'unparsed'),
                       amount numeric NULL, currency, merchant NULL,
                       card_mask, occurred_at timestamptz, balance_after numeric NULL,
                       dedup_hash, status ('pending'|'confirmed'|'dismissed'),
                       transaction_id NULL (FK), created_at
                       UNIQUE (user_id, dedup_hash)

card_account_mappings: id, user_id (FK), card_mask, account_id (FK),
                       UNIQUE (user_id, card_mask)
```

Миграция TypeORM; ORM-сущности регистрируются в **двух** местах: `config/data-source.ts` и `app.module.ts` (известная gotcha проекта).

### 3.6 Env / deploy

- `TELEGRAM_BOT_TOKEN`, `TELEGRAM_WEBHOOK_SECRET`, `TELEGRAM_WEBHOOK_URL` — в `backend/.env.example`
- В `deploy.yml` новые переменные добавить и в `env:`, и в `envs:` whitelist (известная gotcha)
- Бот регистрируется пользователем через BotFather; токен — открытый вопрос на этапе деплоя

## 4. Frontend

### 4.1 Структура (FSD)

- `entities/imported-transaction/` — `api/importedTransactionApi.ts` (camelCase→snake_case), `api/useImportedTransactions.ts` (Vue Query: инбокс, счётчик pending, confirm/dismiss мутации с инвалидацией), `api/queryKeys.ts`, `model/types.ts`
- `features/link-telegram/` — секция в профиле: «Подключить Telegram» → deep link, статус «Подключён как @username», отвязка
- `pages/` — `TelegramInboxPage` (список) и `ImportConfirmPage` (флоу подтверждения); роуты с auth guard

### 4.2 Инбокс

Список pending-импортов (сумма, тип, мерчант, карта, дата). Точка входа — со страницы транзакций: баннер/кнопка «На подтверждение (N)» с бейджем-счётчиком, виден только при N > 0. Пустое состояние — `EmptyState` с инструкцией про бота.

### 4.3 Флоу подтверждения (полноэкранная страница)

В стиле add-transaction, строго по дизайн-системе:

1. **Hero** — сумма (редактируемая; для `balance_change` предзаполнена дельтой или пустая), валюта, тип, мерчант, карта, дата
2. **Счёт** — `AccountSelector`, предзаполнен по `card_account_mappings`
3. **Категория** — `CategoryChips` (категории расходов или доходов по типу)
4. **Долг** — «записать как долг»: выбор человека, вся сумма уходит в долг
5. **Разделить на людей** — переиспользование `split-expense` (SplitExpenseSection)
6. **Сканировать чек** — полный OCR-визард `scan-receipt`; сумма чека сверяется с импортированной, доступно попозиционное деление
7. **Подтвердить** — создаёт транзакцию со всем выбранным (существующие флоу), вызывает `confirm`, инвалидация `invalidateTransactionRelated` (а при долгах — `invalidateDebtRelated`), авто-переход к следующему pending-элементу
8. **Отклонить** — `dismiss`, переход к следующему

Опции 4 и 5 взаимоисключающие (вся сумма в долг ↔ деление); скан чека совместим с делением (механика scan-receipt).

## 5. Безопасность

- Webhook принимает только запросы с корректным `X-Telegram-Bot-Api-Secret-Token`
- Linking-токен одноразовый, TTL 15 минут
- Бот игнорирует всё, кроме личных чатов
- Никакие данные, кроме текста присланных уведомлений, не сохраняются
- Инбокс и все JWT-endpoints отдают только данные владельца (user scoping, как везде в проекте)

## 6. Обработка ошибок

| Ситуация | Поведение |
|---|---|
| Форвард от непривязанного TG-аккаунта | Ответ бота со ссылкой на привязку |
| Сообщение не распознано | Ответ «не смог распознать», raw сохраняется как `unparsed` |
| Дубликат | Молча пропускается, попадает в сводку «дублей N» |
| `balance_change` без предыдущего баланса | Сумма пустая, пользователь вводит в флоу |
| Истёкший/использованный linking-токен | Ответ бота «ссылка устарела, сгенерируй новую в приложении» |
| Ошибка создания транзакции при подтверждении | Импорт остаётся `pending`, ошибка показывается toast'ом, можно повторить |

## 7. Тестирование

- **Парсер — ядро фичи:** юнит-тесты на всех трёх реальных форматах + вариации (`HUMO-CARD`/`HUMOCARD`, `🕓`/`🕘`, отсутствие мерчанта, числа с/без тысячных, граничные суммы)
- Дедупликация: повторный импорт того же текста не создаёт второй записи
- Линковка: валидный/истёкший/повторный токен, повторная привязка
- Дельта для `balance_change`: с предыдущим балансом, без него, отрицательная
- Confirm: маппинг карты сохраняется, статус меняется, транзакция связывается
- Frontend: vitest-тесты композаблов флоу подтверждения (по образцу `useReceiptWizard.spec.ts`)

- Сабагенты — **только Opus**

## 9. Вне объёма v1 (осознанно отложено)

- Автоматический перехват (MTProto-ридер на выделенном техническом аккаунте) — pipeline спроектирован так, что ридер можно добавить вторым входом без переделки
- Другие банки/форматы — добавляются новыми парсерами в реестр по накопленным `unparsed`
- Автокатегоризация по мерчанту (запоминание merchant → category)
- Push-уведомление «у вас N неподтверждённых импортов»
- Лимиты/премиум-гейтинг
